# STEP3 設計：ライセンス偽造を止める

最終更新: 2026-07-01 ／ ステータス: 設計（未実装）

---

## 1. 今の問題（なぜ危ないか）

ライセンスは **会社名という“ただの文字列”** だけで管理されている。秘密の鍵が無い。
配布アプリが公開キーで以下をやっている：

| 操作 | 場所(main.ts) | 何が危ないか |
|---|---|---|
| 自分を自動登録（active/50クレジット） | syncRemoteLicense POST | 誰でも会社名を騙って有効ライセンスを作れる |
| クレジット残数を自分で書き込み | syncCreditsToRemote PATCH | クライアントがクレジットを盛れる（改ざん） |
| プラン/クレジットを読み取り | fetchLicenseByName GET | 他社の契約状況を覗ける |
| 承認/却下 PATCH | remote:approve/reject | 公開キーで他社ライセンスを操作できる |

**核心**：本人確認の仕組みが無い。だから「本人だけができる」を作る必要がある。

---

## 2. 目指す形（設計方針）

### 方針A：ライセンスごとに「秘密の鍵（トークン）」を持たせる ★推奨★
- 各ライセンスに、推測不可能なランダムな **license_token** を発行
- クライアントは初回にトークンを受け取り、ローカルに**暗号化保存**（既存のapi-config暗号化を流用）
- 以降の「確認」「クレジット消費」は、会社名ではなく**トークンで認証**
- remote_licenses への読み書きは **Edge Function（管理キー）だけ**が行う。公開キーは一切触れない

これで「トークンを持つ本人」以外は、読むことも書くこともできなくなる。

### クレジットの扱い（要判断・下記セクション5）
- 現状：クライアントが残数を計算してサーバーに push（改ざん可能）
- 目標：**サーバーがクレジットを減算**（server-authoritative）＝改ざん不可
- トレードオフ：完全サーバー方式にすると**オフライン時に見積もりできなくなる**

---

## 3. 作るEdge Function（案）

| 関数 | 入力 | 動作 | 認証 |
|---|---|---|---|
| **license-register** | company_name | pending状態でライセンス作成＋トークン発行し返す。レート制限あり | なし（新規登録用・乱用対策必須） |
| **license-verify** | license_token | {active, plan, credits, blocked_message} を返す | トークン |
| **license-consume** | license_token, amount | クレジットをサーバー側で減算し新残数を返す。不足/停止なら拒否 | トークン |
| **license-admin** | admin_secret, action, target | 承認/却下/クレジット設定。管理者専用 | 管理者シークレット |

すべて内部で service_role を使用。公開キーでは remote_licenses に触れない。

---

## 4. 最終的なRLS（STEP3完了後）

| テーブル | anon(公開キー)の権限 |
|---|---|
| remote_licenses | **一切なし**（全操作をEdge Function経由に） |
| app_activity | INSERT のみ（現状維持） |
| estimate_feedback | INSERT のみ（STEP2で確定） |
| cost_coefficients | SELECT のみ（STEP2で確定） |

管理ダッシュボードは公開キーをやめ、**管理者シークレット or service_role** で動かす。

---

## 5. ★あなたに決めてほしいこと★

### 決定1：クレジットの厳密さ
- **(a) 完全サーバー方式**：消費のたびにサーバーが減算。改ざん不可。ただし**オフライン時は見積もり不可**。
- **(b) ハイブリッド**：オンライン時はサーバー減算、オフライン時はローカルで暫定消費し次回オンラインで精算。改ざん耐性は中、オフラインOK。
- **(c) 当面ゆるめ**：トークン認証だけ先に入れ、クレジットは今のまま（読取はサーバー、書込停止）。まず「偽造発行」を止めることを優先。
- 👉 建設現場はネット不安定なこともある。おすすめは **(b)** か、まず **(c)** で偽造を止めてから (b) へ。

### 決定2：既存6件のライセンス移行（ここが一番慎重）
現在の顧客（有料のAくん含む）は**トークンを持っていない**。新方式に移す必要がある。
- 案：新バージョン初回起動時に「会社名で1回だけ自分のトークンを受け取る」移行処理を入れる（レガシー経路を移行期間だけ残す）。
- 移行が終わったら、会社名経路を完全に閉じる。
- 👉 **有料顧客の認証を止めないため、移行期間を設けて段階的に切り替える**。

### 決定3：管理ダッシュボードの認証
- service_role キーを起動時に貼る運用（管理者専用・ローカル）でよいか。

---

## 6. 安全な進め方（ロールアウト順序）

```
1. remote_licenses に license_token カラム追加（既存行にもトークン発行）
2. Edge Function 4本を作成・デプロイ（本番では未接続）
3. テスト用ライセンスで verify / consume / register を検証
4. 新バージョンのアプリを作成：
   - 初回にトークンを受け取り暗号化保存（移行処理）
   - verify/consume/register をEdge Function経由に切替
5. 新バージョンをリリース → 顧客に行き渡らせる（移行期間）
6. 移行完了を確認後、remote_licenses の anon 権限を全停止（最終RLS）
7. 管理ダッシュボードを service_role 化
```

⚠️ 6を早まると有料顧客の認証が止まる。**必ず移行完了を確認してから**締める。

---

## 7. リスクと対策

| リスク | 対策 |
|---|---|
| 有料顧客の認証停止 | 移行期間を設け、段階切替。締める前に全顧客の移行を確認 |
| オフラインで使えない | クレジットはハイブリッド(b)方式を採用 |
| register関数の乱用（大量発行） | レート制限＋pending状態で作り、管理者承認を必須に |
| トークン漏洩 | ローカルは暗号化保存。漏れたら管理画面で再発行できるように |
