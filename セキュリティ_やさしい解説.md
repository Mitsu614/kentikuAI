# 建築ブースト セキュリティ対応 やさしい解説

最終更新: 2026-07-01

---

## 0. まず結論（3行）

- いま「データ倉庫（Supabase）」の鍵のかけ方が甘く、**誰でも中身を見たり壊したりできる**状態だった。
- 応急処置（STEP1）は完了。**全削除・盗み見は止めた**。
- でも**ライセンスの偽造**はまだできる。これを止めるのがSTEP2→STEP3。

---

## 1. 登場人物（これだけ分かればOK）

### Supabase（スーパーベース）
建築ブーストのデータを置いている**クラウドの倉庫**。
顧客のライセンス、実績データ、見積もり係数などがここに入っている。

### 鍵は2種類ある
| 鍵 | たとえ | どこにある | 危険度 |
|---|---|---|---|
| **公開キー**（anon / `sb_publishable_...`） | 玄関の「呼び鈴」 | 配布アプリの中に**埋め込まれている**＝誰でも取り出せる | 公開前提。**単体では無力でないといけない** |
| **管理キー**（service_role） | 「マスターキー」 | 自分のPC・サーバーだけ | 絶対秘密。漏れたら終わり |

ポイント：**公開キーはアプリに入っている以上、世界中の誰でも入手できる**前提で考える。
だから「公開キーで何ができるか」を絞り込まないといけない。

### RLS（行レベルセキュリティ）
倉庫の各棚（テーブル）にかける**「誰が・何をしていいか」のルール**。
- RLSが**無い** → 公開キーで全部できる（見る・足す・書き換える・**消す**）
- RLSが**有る** → 「公開キーは“足す”だけ」のように細かく制限できる

今回の問題は、いくつかの棚で**RLSが無かった**こと。
＝呼び鈴を持っているだけの他人が、倉庫の中身を全部見て、書き換えて、消せた。

---

## 2. 何が危なかったか（具体例）

公開キーで実際にアクセスして確認した結果：

| 棚（テーブル） | 入っているもの | 危険 |
|---|---|---|
| **remote_licenses** | 顧客のライセンス・課金情報 | 🔴 **誰でも自分にライセンス発行（タダ使い）／顧客のを削除** |
| app_activity | 利用状況ログ | 🔴 誰でも閲覧 |
| estimate_feedback | 実績データ（匿名） | 🔴 誰でも閲覧・改ざん |
| cost_coefficients | 見積もりの補正係数（AIの頭脳） | 🔴 誰でも書き換え＝**見積もり精度を破壊** |

一番怖いのは **remote_licenses**（課金の根幹）と **cost_coefficients**（AIの精度）。

---

## 3. 対応は3段階

```
STEP1【完了】 応急処置：鍵をかけ直して「全削除・盗み見」を止める
STEP2【準備済】 AIの頭脳(係数)を他人が書き換えられないようにする
STEP3【これから】 ライセンス偽造を止める（一番大事・一番慎重に）
```

---

## 4. STEP1：応急処置（✅完了・実証済み）

### やったこと
全部の棚にRLS（ルール）をかけ直した。公開キーには「いまアプリが実際に使う操作」だけ許可。

具体的には `learning-server/rls-fix.sql` をSupabaseで実行した。

### 結果（数字で確認済み）
- **app_activity** → 公開キーで見えなくなった（`*/0`）＝鍵が効いた証拠
- 係数・ライセンス・実績はアプリ用に読めるまま＝**アプリは壊れていない**
- どの棚も**公開キーでの「削除」が不可**に＝有料顧客のライセンスを消される事故を防止

### 副作用（1つ）
管理ダッシュボードは公開キーで app_activity を読んでいたので、
**ダッシュボードの「利用状況」表示が空になる**（データは無事、見えないだけ）。
→ STEP3でダッシュボードを管理キーに切り替えれば直る。

---

## 5. STEP2：AIの頭脳を守る（準備完了・デプロイ待ち）

### なぜ必要か
STEP1のあとも、公開キーで `cost_coefficients`（係数）を**書き換えられる**状態。
なぜなら、今のアプリは「実績を集計→AIで分析→係数を書き込む」を**アプリ自身（公開キー）でやっている**から。
公開キーに書き込みを許さざるを得ない＝他人も書き込める。

### 考え方（ここが肝）
**「計算と書き込み」をアプリから取り上げ、サーバー（Supabaseの中の小さなプログラム）にやらせる。**
アプリは「再計算して」と**お願いするだけ**。データは渡さない。

こうすると、誰がそのサーバーを呼んでも、
**サーバーは“倉庫の実データ”からしか係数を作らない** → デタラメを注入できない。
＝呼び出せても汚染は不可能。

この「Supabaseの中の小さなプログラム」が **Edge Function（エッジ・ファンクション）**。

### 用意したもの
| ファイル | 中身 |
|---|---|
| `supabase/functions/update-coefficients/index.ts` | サーバー側で係数を計算・保存するプログラム |
| `app/src/main/supabase-sync.ts`（改修済） | アプリは「再計算して」と送るだけに変更 |
| `supabase/functions/update-coefficients/DEPLOY.md` | デプロイ手順書 |
| `learning-server/rls-step2-tighten.sql` | 仕上げに鍵を締めるSQL |

### やる手順（順番厳守）
1. Edge Function をデプロイ（DEPLOY.md の手順1〜4）
2. 公開キーで叩いて `{"updated":N}` が返るか確認（手順5）
3. アプリを**新バージョンでリリース**（通常のアップデート手順）
4. **新バージョンが顧客に行き渡ってから** `rls-step2-tighten.sql` を実行

⚠️ なぜ順番厳守か：4を先に流すと、**旧バージョンの学習更新が止まる**（見積もり自体は動くが、係数が更新されなくなる）。新バージョンが行き渡ってから締める。

### デプロイのコマンド（プロジェクトのフォルダで実行）
```
npx supabase login                                   ← ブラウザでログイン
npx supabase link --project-ref slhgkedzlormaovwpadi ← プロジェクトに接続
npx supabase secrets set ANTHROPIC_API_KEY=sk-ant-…  ← Claudeの鍵をサーバー側に保管
npx supabase functions deploy update-coefficients --no-verify-jwt  ← デプロイ
```
※ このセッションでやるなら各行の頭に `!` を付けて打つ。

## 6. STEP3：ライセンス偽造を止める（最重要・最慎重・これから）

### 問題
配布アプリが**公開キーで remote_licenses を発行・更新**している。
＝他人が公開キーで「自分にライセンス発行」「顧客のを書き換え」できる。

### 直し方（STEP2と同じ発想）
- ライセンスの**発行・更新・認証チェック**を Edge Function（管理キー）に移す
- アプリは「この会社を認証して」「クレジットを使った」と**お願いするだけ**
- 管理ダッシュボードも公開キーをやめて**管理キー**で動かす（管理者専用なので安全）

### なぜ一番慎重にやるか
ここを間違えると、**有料顧客（Aくん）のライセンス認証が止まる**＝即クレーム。
なので：
1. Edge Function を作る
2. テスト用ライセンスで十分に確認
3. アプリをリリースして行き渡らせる
4. 最後に公開キーの発行・更新権限を停止

時間をとって、テストしながら進める。

## 7. よく使う言葉まとめ

| 言葉 | 意味 |
|---|---|
| Supabase | データを置くクラウド倉庫 |
| 公開キー(anon) | アプリに埋め込む鍵。誰でも入手できる前提 |
| 管理キー(service_role) | 絶対秘密のマスターキー |
| RLS | 棚ごとの「誰が何していいか」ルール |
| ポリシー | RLSの個別ルール（例：公開キーは追加だけ可） |
| Edge Function | Supabaseの中で動く小さなサーバープログラム |
| デプロイ | プログラムをサーバーに設置して動かすこと |
